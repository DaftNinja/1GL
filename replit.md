# replit.md

## Overview

This is a **Power Trends** application branded for **1GigLabs** that generates AI-powered power infrastructure analyses for data centre site selection. Users select a country and the system uses OpenAI to generate comprehensive reports covering grid capacity, energy mix, regulatory environment, location suitability, and investor insights.

**Brand:** 1GigLabs (1giglabs.com) - A UK-based provider of managed colocation and connectivity services for IT providers, public institutions, and government organisations. Core values: openness, local focus, and flexibility. Focus on sustainability with 100% green energy and CO2 neutral by 2030 goal.

**Color Scheme:** Blue professional palette (primary: hsl(207, 90%, 54%)).

The application follows a monorepo structure with a React frontend, Express backend, and PostgreSQL database using Drizzle ORM.

## User Preferences

Preferred communication style: Simple, everyday language.

## System Architecture

### Authentication & Security
- **Auth Method**: Email/password authentication (no Replit OIDC)
- **Password Storage**: bcrypt with 12 salt rounds
- **Sessions**: PostgreSQL-backed via connect-pg-simple, stored in `sessions` table
- **Work Email Enforcement**: Personal email domains (gmail, yahoo, hotmail, outlook, etc.) are rejected at registration
- **Protected Routes**: All `/api/*` endpoints require authentication via `isAuthenticated` middleware
- **Auth Routes**:
  - `POST /api/auth/register` — Create account with work email + password
  - `POST /api/auth/login` — Sign in with email + password
  - `POST /api/auth/logout` — End session
  - `GET /api/auth/user` — Get current authenticated user
- **Frontend Auth**: `AuthPage.tsx` shows login/register form; `App.tsx` gates all routes behind auth check
- **User Roles**: `role` field on users table (default: "analyst"), prepared for future RBAC
- **Files**: `server/replit_integrations/auth/replitAuth.ts` (session setup, middleware), `server/replit_integrations/auth/routes.ts` (auth endpoints), `shared/models/auth.ts` (users schema)

### Audit Logging
- **Table**: `audit_logs` in `shared/schema.ts` — tracks userId, userEmail, action, entityType, entityId, metadata (JSONB), ipAddress, createdAt
- **Logged Actions**: LOGIN, LOGOUT, REGISTER, GENERATE_ANALYSIS, VIEW_REPORT, DELETE_ANALYSIS, GENERATE_TAM, GENERATE_POWER_TRENDS
- **API**: `GET /api/audit-logs` — returns recent audit log entries
- **UI**: `/audit-logs` page with activity feed, accessible from UserMenu in navigation
- **Files**: `client/src/pages/AuditLogs.tsx`, `server/routes.ts` (action logging calls)

### Real-Time Collaboration
- **Tables**: `report_comments`, `report_assignments`, `report_activity` in `shared/schema.ts`
- **Backend**: `server/collaboration.ts` — SSE presence stream, comments CRUD, assignments CRUD, activity feed
- **Presence**: In-memory `presenceMap` keyed by analysisId → userId; SSE broadcasts viewer list on connect/heartbeat/disconnect; 30s stale timeout; 15s keepalive
- **Heartbeat**: Client sends POST every 20s to `/api/analyses/:id/presence/heartbeat`
- **SSE Reconnect**: Exponential backoff (1s → 30s max) on connection error; heartbeat paused during disconnect
- **Security**: All routes require `isAuthenticated`; comment delete checks ownership (userId match); assignment update validates analysisId scoping; all params validated with `isNaN` guards
- **Frontend**: `CollaborationPanel` component in Dashboard action bar; presence avatars, comments tab (with section picker), assignments tab (with status transitions), activity tab
- **Storage**: `getReportCommentById`, `getReportAssignmentById` for ownership/scoping checks
- **Files**: `server/collaboration.ts`, `client/src/components/CollaborationPanel.tsx`, `client/src/hooks/use-presence.ts`

### AI Content Labelling
- **Component**: `client/src/components/AIContentLabel.tsx` — reusable label with `banner`, `badge`, and `inline` variants
- **Placement**: Banner variant displayed at top of Dashboard reports, TAM analyses, and Power Trends analyses
- **Export Disclosure**: AI-generated content warning embedded in HTML exports for Dashboard, TAM, and Power Trends
- **Text**: "This report was generated by AI (GPT-4o) on [date]. All data, figures, and insights should be independently verified before use in decision-making."

### Frontend Architecture
- **Framework**: React with TypeScript, built using Vite
- **Routing**: Wouter (lightweight React router)
- **State Management**: TanStack React Query for server state
- **UI Components**: shadcn/ui component library with Radix UI primitives
- **Styling**: Tailwind CSS with CSS variables for theming
- **Animations**: Framer Motion for page transitions and loading states

- **Export Features**: html2canvas and jsPDF for PDF generation, pptxgenjs for PowerPoint
- **User Menu**: `UserMenu` component in navigation bars shows user email, audit log link, and logout button

### Backend Architecture
- **Framework**: Express 5 on Node.js with TypeScript
- **Build Tool**: esbuild for production server bundle, tsx for development
- **API Design**: RESTful endpoints defined in `shared/routes.ts` with Zod schemas for validation

### Database Layer

### Leadership Validation Pipeline
- **Route**: `/verified-executives` — Management UI for verified executive data
- **Schema**: `verified_executives` table (companyName, name, role, sourceUrl, verifiedAt)
- **API**: CRUD at `/api/verified-executives`, `PATCH /api/analyses/:id/leadership` for review confirmation
- **Validation Steps** (applied during company report generation):
  1. **Verified Executives DB Lookup**: Loads confirmed executives from `verified_executives` table and injects into validation prompt
  2. **Web Scraping**: Fetches the company's official leadership page (tries `/about/leadership`, `/leadership`, `/about/team`, investor relations) as authoritative source; SSRF-protected with domain validation and private IP blocking
  3. **AI-Powered Leadership Research**: Supplementary AI call to research current executive team (lower priority than web-scraped data)
  4. **AI Validation Pass**: Cross-references generated leadership against all sources with explicit priority hierarchy: DB verified > website data > AI research > initial generation
  5. **Confidence Scoring**: Each executive rated high/medium/low; verified DB entries and website-confirmed executives get "high"
  6. **Source URLs**: LinkedIn/public source links attached to executives where known
- **User Review Step**: Reports generated with `leadershipConfirmed: false`; Dashboard shows "Pending Review" badge with inline editing panel to correct/confirm executives
- **Dashboard Display**: Confidence badges (green/amber/red), source links, verified/pending status, edit button

### Metric Display
- **MetricValue Component**: Adaptive text sizing for Dashboard metric cards — short values (≤60 chars) render as `text-2xl font-bold font-serif`, longer text renders as `text-base font-medium` with relaxed line-height
- **HTML Export**: Uses corresponding `info-value short`/`info-value long` CSS classes via `infoValue()` helper
- **AI Prompt Guidance**: ESG scores, TVO totals, payback periods, and maturity levels instructed to "KEEP SHORT" with max character limits

### Analysis Generation Pipeline (Parallelized)
- **Architecture**: The main analysis is split into 4 parallel GPT-4o calls + concurrent leadership research, merged with allowlist-filtered keys
  - **Group A**: Core business (companyName, website, colorScheme, executiveSummary, overview, financials, strategy/leadership)
  - **Group B**: Market intelligence (market, SWOT, riskAssessment, growthOpportunities)
  - **Group C**: Technical depth (technicalSpend, ESG, digitalTransformation)
  - **Group D**: Sales enablement (salesEnablement with 1GigLabs-specific positioning)
- **Error Handling**: Uses `Promise.allSettled` — non-critical groups (B, C, D) can fail without crashing the entire request; Group A (core) is required
- **Merge Safety**: Each group's output is filtered through an allowlist of expected keys before merging, preventing key collisions
- **Leadership Pipeline**: Runs sequentially after Group A completes (needs website URL from Group A for web scraping); AI research runs concurrently with all groups
- **Performance**: ~40s total (down from ~4 min with single monolithic GPT-5.1 call)

### Key Design Patterns
- **Shared Types**: TypeScript types and Zod schemas shared between frontend and backend via `@shared/*` path alias
- **API Contract**: Routes defined with method, path, input/output schemas in `shared/routes.ts`
- **Storage Pattern**: Database operations abstracted through storage classes (e.g., `DatabaseStorage`, `chatStorage`)
- **Integration Modules**: Reusable AI integration code organized in `server/replit_integrations/`
- **Schema Field Names**: Internal schema uses `stellanorServices`, `stellanorSolution`, `stellanorService`, `stellanorAdvantage` field names (kept for backward compatibility with existing data); display text uses "1GigLabs" branding

- **SSEN**: Network capacity headroom + DC probability inference. Server module: `server/ssenData.ts`. APIs: `/api/ssen/headroom`, `/api/ssen/dc-probability`. Env var: `SSEN_NERDA_API_KEY`. DC Probability: scores each SSEN substation across 6 signals (contracted demand ≥10 MVA +2, contracted/max ratio ≥1.4 +2, Red RAG with reinforcement works +1, BSP/GSP level +1, diesel ECR assets within 2km +3, within DC geography cluster +2); Grade High ≥6, Medium 4–5; also fetches SSEN ECR 1MW CSV (`ssen_ecr_part_1_1mw_march_2026_csv.csv`) to identify diesel/engine assets registered near each substation. 12 DC clusters defined for SEPD (Reading/Winnersh, Farnborough, Basingstoke, Southampton, Portsmouth, Oxford, Swindon) and SHEPD (Edinburgh Park, Glasgow, Dundee, Perth, Aberdeen). Purple building icons on map (dark=High, light=Medium). 24h cache.
- **UKPN**: Data centres, grid/primary sites, and DFES headroom. Server module: `server/ukpnData.ts`. APIs: `/api/ukpn/datacentres`, `/api/ukpn/grid-primary-sites`, `/api/ukpn/dfes-headroom`. Env var: `UKPN_API_KEY`. DFES Headroom: 981 substations (LPN London + EPN Eastern + SPN South Eastern) from `dfes-network-headroom-report` dataset; "Electric Engagement" scenario; demand headroom 2025/2030/2035 + gen inverter + gen synch headroom 2025; RAG Green ≥15 MW, Amber 5–15 MW, Red <5 MW.
- **NPg (Northern Power Grid)**: Transformer utilisation and connection queue data from NPg OpenDataSoft portal. Server module: `server/npgData.ts`. APIs: `/api/npg/utilisation`, `/api/npg/connection-queue`, `/api/npg/ndp-headroom`. Env var: `NPG_API_KEY` (utilisation only; connection-queue and ndp-headroom are public). 24-hour in-memory cache. Covers NE England & Yorkshire. NDP Headroom: 626 substations (NPgY + NPgN) from `npg_ndp_demand_headroom` + `npg_ndp_generation_headroom` datasets; "NPg Reference Scenario"; demand/gen headroom 2025–2035; RAG thresholds Green ≥15 MW, Amber 5–15 MW, Red <5 MW.
- **ENW (Electricity North West)**: BSP and PRY network headroom data from ENW OpenDataSoft portal. Server module: `server/enwData.ts`. API: `/api/enw/headroom`. Env var: `ENW_API_KEY`. 12-hour in-memory cache. Fetches 68 BSP + 367 PRY sites in parallel. Fields: `dem_hr_firm_mw`, `dem_hr_non_firm_mw`, `gen_hr_inverter_mw`, `gen_hr_synchronous_mw`, `batt_storage_hr_mw`. Band thresholds: Green ≥20 MW, Amber 5–20 MW, Red <5 MW. Covers NW England.

### DC Insights Map Layer (1GigLabs Validated Dataset)
- **Overview**: Pan-European data centre map layer using the 1GigLabs validated dataset. 2,764 geo-validated records across Europe: 1,796 Operational, 596 Planned, 299 Construction, 64 Land Bank, 9 Prospective Expansion. All records pass geo-validation (confirmed on land).
- **Primary data source**: Static JSON file bundled with the server — `server/data/dc-insights.json`. Extracted from `dc-insights-1GL-validated_4.xlsx` (analyst-validated). Fields: id, name, lat, lng, geo_region, operator, capacity_mw, tier, validation, validation_notes, geo_checked.
